# ROS 1 remote-access spike

Feasibility spike for a ROS 1 Noetic `foxglove_bridge` with remote access
(LiveKit/WebRTC via the SDK's `RemoteAccessGateway`), running in Docker.

## The problem

The SDK's remote access support requires glibc >= 2.35 on Linux (see
`ros/src/foxglove_bridge/CMakeLists.txt`). ROS 1 Noetic is only released for
Ubuntu 20.04 (focal, glibc 2.31). This spike proves the workaround: run Noetic
*built from source* on Ubuntu 22.04 (jammy, glibc 2.35).

## What was proven

1. **`Dockerfile`** (stage A): the C++ SDK with `REMOTE_ACCESS ON` builds, links,
   and runs on a minimal jammy image. The gateway authenticates against a local
   Foxglove API server and reaches `connected`.
2. **`Dockerfile.noetic`** (stage B): a minimal Noetic stack (`ros_comm` +
   `topic_tools`, generated by `rosinstall_generator`) builds from source on
   jammy and runs (roscore, rostopic pub/echo work on Python 3.10). A single
   binary links both roscpp and the remote-access SDK, subscribes to `/chatter`
   via `topic_tools::ShapeShifter`, and forwards raw serialized bytes to a
   `ros1`-encoded `RawChannel` whose schema comes from the connection header's
   message definition — no message definition cache needed, unlike ROS 2.

## Build

From the repo root:

```sh
make build-cpp-dist                  # jammy-built SDK dist with remote-access (once)

# Stage A: gateway-only hello world
docker build -f ros/ros1_spike/Dockerfile -t ros1-spike .

# Stage B: Noetic-on-jammy + mini ROS 1 bridge
docker build -f ros/ros1_spike/Dockerfile.noetic -t ros1-spike-noetic .
```

## Run

With the local stack up (LiveKit compose, API server — see the remote-access
project notes):

```sh
docker run --rm --network host \
  -e FOXGLOVE_DEVICE_TOKEN=<token> \
  -e FOXGLOVE_API_URL=http://localhost:3000/api \
  ros1-spike-noetic
```

The entrypoint starts an in-container roscore and a 5 Hz `std_msgs/String`
talker on `/chatter`. In the app: Devices -> (device for the token) -> Connect;
`/chatter` and `/spike/timestamp` should be live.

## Jammy build notes (inputs for the real ROS 1 bridge)

- `rosconsole` does not compile against jammy's log4cxx 0.12 (`LoggerPtr` became
  a `std::shared_ptr`). Built with `-DROSCONSOLE_BACKEND=print` instead, which
  drops the log4cxx dependency entirely; stdout logging is right for containers.
- `class_loader`/`pluginlib` (pulled in via rosbag) need `libconsole-bridge-dev`,
  `libpoco-dev`, `libtinyxml2-dev`.
- ROS python build tooling comes from PyPI (no jammy/noetic apt repo); `empy`
  must be pinned `< 4` for genmsg.
- `rostopic` imports rosbag, which needs `pycryptodomex` and `python-gnupg` at
  runtime.
